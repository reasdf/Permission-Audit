# 越权识别 Skill（分析 .har 版）

> 适用范围：目前只能识别垂直越权漏洞。

## 执行流程

### 1. 运行分析脚本

```bash
python3 {{SKILL_DIR}}/analyze.py <har文件路径>
```

> 📌 将 `{{SKILL_DIR}}` 替换为该 skill 的实际安装目录（即 `SKILL.md` 所在目录）。

脚本会自动：
- 在 HAR 同目录创建 `xxx-analyze-work/` 工作目录
- 执行渐进式越权判定（状态码 → 响应长度 → 阻断标志检测）
- 生成 `xxx-垂直越权报告.md`（含响应预览，最大 2000 字符）

### 2. 大模型深度危害评估

脚本判定为 **确认越权** 的接口，LLM 根据报告中响应预览评估危害等级（高危/中危/低危）。

> 脚本输出的「数据特征」仅为响应数据量的粗略描述（含敏感字段/数据量较大/低危），**LLM 必须根据实际数据内容重新定级**，不以数据量大小为唯一依据。

脚本判定为 **疑似越权** 的接口，LLM 需对每个接口做**二元判定**：是真正的越权漏洞，还是业务层已阻断（正常过滤）。

**评估维度（确认越权接口）：**

| 维度 | 评估内容 |
|------|---------|
| 数据类型 | 是否包含个人信息、业务数据等 |

**疑似越权研判规则（按优先级）：**

| 判定条件 | 结论 | 理由 |
|---------|------|------|
| 低权限响应为 `[]`（空数组），高权限有实际数据 | ✅ 已阻断 | 后端做了数据权限过滤，无权限时返回空 |
| 低权限响应仅含**单项/少量数据**，高权限返回**全量列表** | ✅ 已阻断 | 按用户权限范围做了业务层过滤（如仅返回自己有权限的国家/菜单） |
| 低权限响应为 `{"status":"failure",...}` 或类似业务错误 | ✅ 已阻断 | 业务层无阻断关键词，但实际已拒绝（如 `resultCode: 20009 "No authorized stations"`） |
| 高权限和低权限均返回阻断信息（如 Access Denied） | ✅ 已阻断 | 原始请求本身对该用户也无授权 |
| 低权限返回的数据与高权限**实质性相同**（同量级同类型业务数据） | 🔴 确认为越权 | 应为真正的权限校验缺失 |

**输出格式（确认越权）：** `<危害等级> — <简洁原因>`

- `🔴 高危 — 泄漏全量司机身份证号、手机号`
- `🟡 中危 — 泄漏员工列表（姓名、工号、岗位类型）`
- `🟢 低危 — 仅返回空数组或字典配置项`

**输出格式（疑似越权）：** `✅ 已阻断 — <原因>`

- `✅ 已阻断 — 低权限返回空数组（业务层数据隔离）`
- `✅ 已阻断 — 仅返回有权限的单国家/单菜单（业务层范围控制）`
- `✅ 已阻断 — 响应 status:failure（业务层拒绝）`

**最终修正统计：** 有越权的接口计入最终总数；判定为已阻断的从疑似列表中移除。修正后的最终数量填入报告统计表。

### 3. LLM 改写最终报告

完成深度评估后，LLM 将**以脚本生成的原始报告为基础，重写为最终版本**，包含以下变更：

| 变更项 | 说明 |
|--------|------|
| 报告头统计 | 更新为 LLM 修正后的值（如疑似越权归入已阻断后数量归零） |
| 危害等级 | 脚本初判的「含敏感字段/数据量较大/低危」替换为 LLM 评定的「高危/中危/低危」 |
| 确认越权 Details 区 | 包含 LLM 评估结论与危害等级，以及高/低权限**响应体预览**（按实际内容篇幅截取），不包含原始请求体 |
| 疑似越权章节 | 替换为「✅ LLM 疑似越权复核」章节，逐条说明判定理由 |
| 修正后统计表 | 填充最终数值 |
| 优先修复建议 | 补充 LLM 评估的业务风险说明 |
| 风险 URL 复制区 | 按 P0/P1/P2 重新组织 |

最终报告结构如下：

```
## 🔴 确认越权漏洞（N 个）

### ⛔ 高危（X 个）
| # | 接口 | 方法 | 响应长度 | LLM 评估 |
...
<details>...</details>

### 🟡 中危（Y 个）
### 🟢 低危（Z 个）

## ✅ LLM 疑似越权复核（M 个→全部归入已阻断）

| # | 接口 | 高权限 | 低权限 | LLM 判定 |
...
<details>...</details>

## 📊 修正后统计
## 🎯 优先修复建议（P0/P1/P2）
## 📋 风险 URL 快速复制区（P0/P1/P2）
```

## 补充信息

### 请求配对规则

1. 按 `Method + URL Path + Body（前 500 字符）` 分组
2. 每组分离：**原始请求**（tags 不含 `[重发]`）和 **重发请求**（tags 含 `[重发]`）
3. 无重发请求的组跳过

### 渐进式判定流程

| Step | 检查项 | 判定 |
|:----:|--------|------|
| 1 | 状态码比对 | 不同 → 已阻断 |
| 2 | 响应长度差异率 | 计算但不直接判定 |
| 3 | 阻断标志检测 | 分层匹配（短语/长词/substring/结构感知正则） |
| 4 | 最终矩阵 | 状态码相同 + 差异率≤5% + 无阻断标志 → **确认越权**；状态码相同 + 差异率>5% + 有阻断标志 → **已阻断**；其他 → **疑似越权** |

### 阻断标志检测

| 组 | 策略 | 关键词 |
|:--:|------|--------|
| A — 短语 | substring | access denied, permission denied, 权限不足, 未授权, 拒绝访问 等 |
| B — 长词 | substring | unauthorized, forbidden, restricted, 无权 |
| C — 状态码 | JSON 精确匹配 | 403, 401 |
| D — 结构感知 | 正则匹配语义位置 | error(非空/true), failure, fail, denied, `"success": false` |

### 过滤规则

- 静态资源（`.js`, `.css`, `.png` 等）
- 公共接口（login, logout, register, captcha, verify）
